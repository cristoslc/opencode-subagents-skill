# Synthesis — opencode runtime integration

A skill that lets a host CLI (Claude Code, Codex, Gemini, Cursor, etc.) dispatch subagents through opencode needs to commit to: (1) a transport, (2) a session/CWD model, (3) a handoff format, (4) a stall-and-error contract, (5) a live-steering story. This synthesis distills what the sources say about each.

## 1. Transport

opencode exposes four transports. Three are real candidates; one is wrong-shape.

- **`opencode run`** — one-shot, CLI-only, in-process server, exits when SSE sees `session.status: idle`.
- **`opencode serve` + HTTP/SSE (or SDK)** — long-lived daemon, REST + SSE, multi-client.
- **`opencode acp`** — JSON-RPC over stdio. Designed for editor integration, not subagent dispatch.
- **MCP** — opencode is an MCP **client** only; cannot be the dispatch target.

**Recommendation.** Support both `opencode run` (stateless, simple) and `opencode serve` + REST (warm, fast, multi-task). Skip ACP — it adds a protocol layer that buys nothing for non-editor hosts. Skip MCP-as-target — not implemented upstream.

The two supported modes have different tradeoffs:

| | `run` | `serve` + REST |
|---|---|---|
| Cold-start cost | High (process + model load every call) | Pay once |
| Concurrency model | One process per task | One server, N sessions |
| Live steering | `--share` URL only | TUI-control endpoints + multi-attach |
| Permission relay | Auto-deny `question`/`plan_*`; otherwise needs `--dangerously-skip-permissions` | Same — plus a known relay bug under `serve+attach` (issue #16367) |

For a first version of the skill, `opencode run` per task is the simpler design. `serve` mode is the second milestone.

## 2. Session and working-directory model

The sources are unanimous on one point: **opencode binds a session to a working directory**, and shared-CWD parallelism races (issue #4251). Baton, awesome-opencode wrappers, and the field retro all converge on the same coordination primitive: **one worktree per task**.

But — and this is the design-critical wrinkle — **host harnesses differ in how (and whether) they can change working directory mid-session.** Claude Code can shift between worktrees within a single session via `cd` and tool-routing logic. Codex and Gemini are not known to do this safely; their tool execution may be pinned to the launch CWD. So the skill cannot assume the parent will manage worktrees correctly on its own.

Implication: the skill must own working-directory selection deterministically, independent of host capability.

- The dispatch entrypoint takes an explicit, absolute `--cwd` (or worktree path) argument; defaults are forbidden.
- Before launching opencode, the skill **verifies** that the path exists, is a git work tree, and matches the expected branch/worktree label. Verification fails closed.
- The opencode session is launched with `--dir <verified-path>` (and `--attach <server>` when serve mode is used; the same `--dir` then names the *remote-side* path).
- Outputs (logs, JSON event stream, edited file list) carry the verified CWD as a tag, so the operator can confirm out-of-band that the agent worked where expected.

## 3. Handoff format — on-disk scripts, not inline tool calls

The naive design is "host parent calls a single dispatch tool with a prompt argument inline." That works for a one-off, but it's brittle: prompts get long, contain quotes, must be regenerated by hand for retries, and don't survive across sessions.

The durable pattern, picked up from research-keeper's `shared-decisions-vN.md` workflow and from how multi-round iteration actually works in practice: **render a per-handoff script to disk, then execute it.** Each handoff is a standalone artifact: prompt, files, model, agent, CWD, env all materialized as a file the operator can read, edit, replay, or hand to a colleague.

Mechanics:

- Templates live under the skill (e.g., `templates/dispatch.sh.j2`) and render via Jinja2 (`j2`) or an equivalent.
- One template per dispatch *kind* (e.g., `single-file-fix`, `parallel-review-fanout`, `headless-spike`), not one template global to everything.
- The rendered artifact is written to a per-task directory (e.g., `.opencode-dispatch/<task-id>/dispatch.sh` plus `prompt.md`, `files.txt`, `env`).
- The host parent's "tool call" reduces to: render template → write to disk → exec the script → tail the output.
- The script is the source of truth for what was dispatched. Replays reuse it. Audits read it.

This also makes parallel-fanout cleaner: render N scripts under N task-id directories, exec them in parallel, collect their outputs at known paths.

## 4. Stall and error contract

Six known headless-stall modes are documented (issue #17516, #16367, #13946, #8203, #1329, #10012); the field retro adds two more (silent stall on glm-5.1, `</think>` token leakage). A dispatcher must handle them — opencode does not.

Defensive defaults the skill should bake in:

- Wrap every `opencode run` invocation in `timeout` with a per-task SLA. Don't trust process exit alone.
- Set `OPENCODE_DISABLE_AUTOCOMPACT=true` for headless dispatch (avoids the silent-exit-on-compaction class).
- Set `OPENCODE_DISABLE_AUTOUPDATE=true` for unattended runs.
- Set `--dangerously-skip-permissions` AND `permission: { "*": "allow" }` in config for non-interactive use.
- Validate post-run: did the agent edit files? Is the working tree dirty in the expected paths? Did the JSON event stream end with `session.status: idle`? Treat any "no" as a stall and respawn or escalate.
- Strip `</think>` blocks from captured stdout before persisting.
- Don't trust frontmatter immutability — diff frontmatter post-run and revert drift mechanically.

For Kimi K2 specifically: route through `@ai-sdk/openai-compatible` (not the built-in `openrouter` provider) or hit Moonshot/Groq directly.

## 5. Live steering and inspection

The advertised win for routing through opencode (vs. each host's built-in subagent runtime) is operator visibility. The skill should make this real, not nominal:

- **In `run` mode** — surface the `opncd.ai/s/<id>` share URL in the script's stdout so the operator can drop into a viewer. Optionally pin `--share auto` in config.
- **In `serve` mode** — print the local URL and the session ID. The operator can `opencode attach http://localhost:4096 --session <id>` to drive the live TUI. The `/tui/append-prompt` and `/tui/submit-prompt` endpoints let the dispatcher *or* the operator inject prompts into a session-in-progress.
- **In both modes** — keep the on-disk handoff script and prompt files around after the run. Re-launch from the same artifacts is a one-liner.

## 6. Provider configuration

Model strings are `provider/model` with optional `:variant` suffix. The most-used providers in this design context:

- `anthropic/`, `openai/`, `google/` — first-class.
- `ollama/` (local), `ollama/<model>:<size>-cloud` (Ollama Cloud).
- `openrouter/` — built-in but flaky for some open-weight models; prefer `@ai-sdk/openai-compatible` for those.
- Custom OpenAI-compatible providers via `@ai-sdk/openai-compatible` or `@ai-sdk/openai`.

Auth lives in `~/.local/share/opencode/auth.json`. Project config in `opencode.json`. `OPENCODE_CONFIG_CONTENT` exists for inline config but **does not interpolate `{env:...}` tokens** — pre-interpolate or use a real file.

## 7. Where this leaves the SKILL.md

The skill should:

- Default to **`opencode run` per task**, with **`opencode serve` + SDK** as a documented opt-in for warm dispatch.
- Require a verified, absolute `--cwd` for every invocation; refuse to launch otherwise.
- Render dispatch as **on-disk scripts from templates**, not inline tool args.
- Bake in the timeout + auto-compaction-off + auto-update-off + permissions-allow defaults.
- Surface the share/attach URL on every run for live inspection.
- Provide a **post-run validator**: stream ended at idle, files edited, frontmatter undrifted, `</think>` stripped.
- Document the failure-mode mitigations explicitly so users who hit them find the fix in one place.

Open questions for the design phase:

1. Do we ship a per-host adapter (Claude Code, Codex, Gemini) that knows how to invoke the rendered script, or a single neutral entrypoint that hosts call uniformly?
2. Do we vend the templates inside the skill, or expect the consumer project to author them with sensible defaults?
3. Where do task-id directories live — under `.opencode-dispatch/` in the consumer repo, or somewhere the skill can clean up automatically?
4. Cleanup policy for completed dispatches (keep artifacts forever, garbage-collect after N days, etc.)?

These are good first SPECs.

## Points of agreement across sources

- HTTP API is the architecture's contract surface; the TUI is just one client. Driving opencode from outside is supported by design.
- Worktree-per-task is the de-facto coordination primitive in production usage.
- `--dangerously-skip-permissions` is required for unattended dispatch.
- Headless reliability requires external timeout wrapping; opencode does not self-heal stalls.

## Points of disagreement / gaps

- No source benchmarks `run` vs `serve` cold-start cost — operators will need to measure.
- ACP is presented in docs as a general agent protocol, but in practice opencode only implements it as the agent side and only editors are documented as clients. There is no canonical example of a non-editor parent driving opencode over ACP.
- Structured-output (`format: { type: "json_schema" }`) is documented in the SDK but not in the CLI; behavior parity is unverified.
- Cross-runtime worktree behavior (Claude Code mid-session `cd` vs Codex/Gemini fixed-CWD) is not directly addressed in any source — it's an inference from harness docs and field experience that the skill must defend against.

## Field-evidence cross-reference

The retro at `~/Documents/code/research-keeper/docs/swain-retro/2026-04-25-multi-round-design-iteration.md` is the most concrete in-the-wild record of this pattern: 4 parallel `opencode run` background processes, ollama-cloud models, per-file ownership, shared-decisions docs as the inter-agent convergence anchor. It validates the worktree-per-task isolation claim (zero merge conflicts across 36 agent-rounds) and surfaces the silent-stall and frontmatter-drift modes that the skill needs to handle.
